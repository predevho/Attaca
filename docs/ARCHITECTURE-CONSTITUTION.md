# ARCHITECTURE-CONSTITUTION

아키텍처 핵심 원칙. 이 문서의 원칙은 임의로 변경하지 않으며, 변경이 필요하면 기록하거나 보고한다.

---

## 1. 서비스 개요

* Attaca는 음악인들을 위한 커뮤니티 SNS다.
* 서로 모르는 음악인들이 자유롭게 소통하고, 자신의 연주회를 소개하며, 구인/구직으로 함께 연주할 사람을 찾는 공간이다.
* FE(웹, 추후 모바일 앱)와 BE(API 서버)는 완전히 분리된다.

---

## 2. FE/BE 분리 원칙

* FE와 BE는 REST API와 WebSocket으로만 통신한다.
* BE는 특정 FE 플랫폼에 종속되지 않는다. 웹으로 시작하되, 동일한 API를 모바일 앱이 그대로 재사용할 수 있어야 한다.
* BE는 화면(뷰) 로직을 갖지 않는다. 표현은 전적으로 FE의 책임이다.

---

## 3. 도메인 구성

Attaca는 다음 도메인으로 구성된다. 각 도메인은 독립된 책임을 갖는다.

| 도메인 | 책임 |
|---|---|
| MEMBER | 회원가입/로그인(JWT + OAuth2), 프로필, 권한(ROLE_USER / ROLE_ADMIN) |
| VERIFIED-PERFORMER | 인증 연주자 신청 → 어드민 승인/거절, 어드민 직접 지정, 인증자 공개 프로필 게시 |
| FEED | 자유 게시글, 피드 타임라인, 댓글, 좋아요 |
| PERFORMANCE | 연주회/공연 등록·홍보 (일시·장소·프로그램 등 구조화된 개체) |
| RECRUITMENT | 구인/구직 공고 (연주자를 찾는 쪽 ↔ 참여를 희망하는 쪽 양방향) |
| CHAT | 1:1 / 1:N 실시간 채팅 |

* 어드민은 별도 도메인이 아니라 MEMBER의 역할(ROLE_ADMIN)로 표현한다.
* 관리 기능(예: 인증 연주자 승인)은 각 도메인 안에서 ROLE_ADMIN 권한으로 수행한다.

---

## 4. 도메인 경계 원칙

* 각 도메인은 독립적으로 개발·테스트할 수 있어야 한다.
* 도메인 간에는 엔티티를 직접 참조하지 않고, 서비스 계층을 통해서만 협력한다.
* 한 도메인이 다른 도메인의 데이터를 "보여주는 것"(뷰 노출)과 그 데이터를 "소유·관리하는 것"은 별개다. 소유·관리 책임은 항상 해당 도메인에 있다.
  * 예: 연주회(PERFORMANCE)가 피드(FEED)에 카드로 노출되더라도, 연주회 데이터의 소유·관리는 PERFORMANCE 도메인에 있다. FEED는 서비스 계층을 통해 참조만 한다.
* 도메인 문서(DOMAIN-XXX-CONSTITUTION.md, DOMAIN-XXX-STATUTE.md) 없이 해당 도메인의 기능을 구현하지 않는다.

---

## 5. 계층 원칙

* 의존 방향은 단방향이다: Controller → Service → Repository.
* 계층을 건너뛰지 않는다. (예: Controller가 Repository를 직접 호출하지 않는다.)
* Entity는 컨트롤러 밖으로 노출하지 않는다. 요청/응답은 DTO로 주고받는다.

---

## 6. 품질 원칙

* 가능하면 TDD를 수행한다.
* 모든 테스트와 린트가 통과해야 작업 완료로 판단한다.
* 에이전트가 생성한 코드도 사람이 이해할 수 있는 구조로 작성한다.
* 요청된 작업 범위를 벗어난 기능을 임의로 추가하지 않는다.
