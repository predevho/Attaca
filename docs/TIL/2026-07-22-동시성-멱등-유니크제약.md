# TIL — 좋아요를 멱등하게: check-then-act 레이스와 유니크 제약 최종 방어선

> 작성일: 2026-07-22
> 목적: FEED 좋아요가 동시 이중요청에도 500이 아니라 200을 유지하도록 만든 방법을 **이 문서 하나로** 이해한다. TOCTOU 레이스, 유니크 제약, `saveAndFlush` + `DataIntegrityViolationException` catch.
> 관련 코드: `domain.feed.service.FeedLikeService#likePost/#likeComment` / 규칙: `docs/DOMAIN-FEED-STATUTE.md §4`

---

## 1. "멱등"이 왜 필요한가

좋아요는 **몇 번 눌러도 결과가 같아야** 한다(눌린 상태 1개). 이런 연산을 멱등(idempotent)이라 한다. 사용자가 따닥 두 번 누르거나, 네트워크 재시도로 요청이 중복되어도 오류가 나면 안 된다.

---

## 2. 순진한 구현의 함정 — check-then-act 레이스

처음 떠올리는 구현:

```java
if (!postLikeRepository.existsByMemberIdAndPostId(memberId, postId)) {
    postLikeRepository.save(PostLike.create(memberId, postId));  // 없으면 저장
}
```

단일 요청이면 잘 된다. 그러나 **동시에 두 요청**이 오면(TOCTOU: Time-Of-Check to Time-Of-Use):

```
요청A: exists? → false          요청B: exists? → false     (둘 다 아직 없음)
요청A: INSERT (성공)            요청B: INSERT (유니크 위반!)
```

두 요청이 모두 `exists` 검사를 통과한 뒤 둘 다 INSERT를 시도한다. `unique(member_id, post_id)` 제약 때문에 두 번째 INSERT가 터지고 → `DataIntegrityViolationException` → 전역 핸들러의 catch-all에 걸려 **500**이 된다. 멱등이어야 할 연산이 서버 오류를 뱉는다.

---

## 3. 해결 — 유니크 제약을 "최종 방어선"으로, 위반은 무해 처리

핵심 발상: **애플리케이션의 exists 검사로는 레이스를 못 막는다. 진짜 방어선은 DB 유니크 제약이다.** 그러니 제약 위반을 오류가 아니라 "이미 눌림"으로 받아들이면 된다.

```java
@Transactional
public void likePost(Long memberId, Long postId) {
    requireActivePost(postId);
    if (postLikeRepository.existsByMemberIdAndPostId(memberId, postId)) {
        return;                                   // 빠른 경로: 이미 눌림 → no-op
    }
    try {
        postLikeRepository.saveAndFlush(PostLike.create(memberId, postId));
    } catch (DataIntegrityViolationException e) {
        // 동시 이중요청 — 유니크 제약이 막아줌. 멱등이므로 무해 처리(200 유지)
    }
}
```

- `exists` 검사는 **성능 최적화**(대부분의 경우 예외 없이 끝남)이지 정확성 보장이 아니다.
- 정확성은 `try/catch`가 보장한다. 레이스로 두 번째 INSERT가 와도 잡아서 no-op.

---

## 4. 왜 `save`가 아니라 `saveAndFlush`인가 (중요)

이게 이 TIL의 진짜 포인트다.

JPA `save()`는 INSERT를 **즉시 실행하지 않는다.** 영속성 컨텍스트에 쌓아뒀다가 트랜잭션 커밋 시점에 flush(=실제 SQL 전송)한다. 그러면 유니크 위반은 **커밋할 때** 터지는데, 그때는 이미 `try/catch` 블록을 벗어난 뒤라 예외를 못 잡는다 → 여전히 500.

`saveAndFlush()`는 그 자리에서 INSERT를 DB로 보낸다. 그래서 유니크 위반이 **try 블록 안에서** 발생하고, catch가 잡을 수 있다.

> 규칙: "제약 위반을 그 자리에서 잡아 처리"하려면 반드시 `saveAndFlush`(또는 명시적 flush)로 예외 발생 시점을 try 안으로 끌어와야 한다.

---

## 5. 취소(unlike)는 왜 그냥 되나

취소는 delete-if-present라 **자연스럽게 멱등**이다:

```java
public void unlikePost(Long memberId, Long postId) {
    postLikeRepository.deleteByMemberIdAndPostId(memberId, postId);  // 없으면 0건 삭제, no-op
}
```

없는 걸 지워도 무해(0 rows affected). 그래서 취소엔 특별한 처리가 없다. 멱등이 원래 쉬운 쪽(삭제)과 어려운 쪽(삽입=유니크 충돌 가능)을 구분하는 게 포인트.

---

## 6. 한 줄 정리

- 멱등 삽입에서 `exists` 검사만으론 동시성 레이스(TOCTOU)를 못 막는다.
- 진짜 방어선은 **DB 유니크 제약**. 위반 예외를 잡아 no-op으로 처리하면 200 멱등이 완성된다.
- 단, `save`는 커밋 때 flush돼서 catch를 못 잡는다 → **`saveAndFlush`로 예외를 try 안으로** 끌어와야 한다.
- 삭제(취소)는 delete-if-present라 원래 멱등.
