# TIL — 도메인 간 참조: ID 참조 vs 엔티티 연관, 그리고 왜 fetch join이 아니라 배치 조회인가

> 작성일: 2026-07-17
> 목적: FEED 도메인 설계 중 나온 질문 — "댓글/글 작성에서 MEMBER를 ID로 참조해 조회하는 게 일반적 아닌가? 문서가 다른 도메인 침범을 막나?" 를 **이 문서 하나로** 이해할 수 있게 정리한다.
> 관련 규칙: `docs/ARCHITECTURE-CONSTITUTION.md §4`, `docs/DOMAIN-FEED-STATUTE.md §5`
> 관련 코드(선례): `domain.member.entity.MemberProfile`(도메인 내부 연관), `domain.verifiedperformer`(도메인 간 ID 참조 협력)

---

## 1. 질문을 정확히 쪼개기

"작성자를 ID로 참조해 조회하는 게 일반적 아니냐"는 맞는 직관이다. **핵심은 "ID로 참조하느냐"가 아니라 "무엇으로 참조하느냐"** 다. 두 개를 헷갈리면 안 된다.

| 방식 | 코드 | 무엇을 저장하나 |
|---|---|---|
| **엔티티 연관(association)** | `@ManyToOne private Member author;` | 다른 엔티티에 대한 **객체 참조**(FK 매핑). `post.getAuthor().getNickname()` 가능 |
| **ID 참조(reference by id)** | `private Long authorId;` | 그냥 **식별자 값**. 닉네임이 필요하면 그 도메인 서비스로 따로 조회 |

우리가 FEED에서 `Post.authorId : Long`을 쓰는 건 **ID 참조**다. 즉 질문자의 직관대로 "ID로 참조"하고 있는 게 맞다. 막는 것은 ID 참조가 아니라 **엔티티 연관**이다.

---

## 2. 문서가 막는 것 — 도메인 "간" 엔티티 침범

`ARCHITECTURE-CONSTITUTION §4`:

> "도메인 간에는 엔티티를 직접 참조하지 않고, 서비스 계층을 통해서만 협력한다."
> "한 도메인이 다른 도메인의 데이터를 '보여주는 것'과 그 데이터를 '소유·관리하는 것'은 별개다."

정리하면 규칙은 딱 하나다:

> **도메인 경계를 넘는 참조(cross-domain)는 `@ManyToOne 다른도메인엔티티`를 쓰지 말고, `Long id` + 서비스 협력으로 해라.**

이유:
* **독립 개발·테스트**: FEED 테스트가 MEMBER 엔티티/테이블을 끌고 오지 않아도 된다.
* **경계 유지**: 객체 그래프가 도메인 경계를 넘어 엉키지 않는다(한 엔티티에서 `.getAuthor().getProfile().get...`으로 아무 도메인이나 타고 들어가는 걸 원천 차단).
* **분리 가능성**: 훗날 서비스 분리(다른 DB/모듈) 시 FK가 경계를 넘지 않아 갈라내기 쉽다.

트레이드오프: 도메인 간 **FK 무결성**과 **fetch join 편의**를 포기한다. 대신 서비스 계층에서 조립(배치 조회)한다.

---

## 3. 중요한 뉘앙스 — 도메인 "내부"는 연관을 써도 된다

이 규칙은 **도메인 간**에만 적용된다. **같은 도메인 안**이면 엔티티 연관이 정상이고 실제로 이 코드베이스도 쓴다.

* 선례: `MemberProfile`은 같은 MEMBER 도메인의 `Member`를 `@OneToOne`으로 연관한다.

그래서 FEED **내부**의 `Comment→Post`, `PostLike→Post`는 선택지가 열려 있다:
* **Long ID 유지(채택)**: 피드 전체가 id 배치·커서·soft-delete 필터로 돌아가므로 카운트/N+1 처리가 id 기반으로 일관되고 lazy 로딩 함정이 없다.
* JPA 연관: `MemberProfile` 선례와 맞고 도메인 내부 FK 무결성을 얻지만, 카운트/커서 로직에서 lazy 로딩을 신경 써야 한다.

→ Attaca FEED는 **내부도 Long ID로 통일**하기로 했다(2026-07-17). "도메인 간은 무조건 ID, 도메인 내부는 팀 판단"이 규칙의 정확한 범위다.

---

## 4. 그럼 닉네임은 어떻게 붙이나 — 서비스 협력 + 배치 조회

`Post.authorId`만 있으니, 응답에 작성자 닉네임/인증뱃지를 채우려면 MEMBER에게 **물어봐야** 한다. 피드 목록은 게시글이 여러 개라 여기서 **N+1**이 문제가 된다.

* 나이브: 게시글 20개 → 작성자 조회 20번(= 1 + N).
* 배치: 게시글 20개의 `authorId` 집합 → **한 번에** `findByIdIn(ids)` → `Map<Long, MemberDisplay>` 로 조립.

`DOMAIN-FEED-STATUTE §5`에서 확정한 방식:
```
MemberQueryService.findDisplaysByIds(Set<Long> ids) → Map<Long, MemberDisplay{nickname, verified}>
```
페이지당 쿼리 수가 고정된다: 게시글 1 + 작성자맵 1 + 좋아요수 1 + 댓글수 1 + 내좋아요 1.

---

## 5. 왜 fetch join이 아니라 배치 조회인가 (핵심)

"N+1이면 fetch join 아니냐"는 반사적 답이 여기선 **틀린다**. 두 가지 이유:

**(1) fetch join 대상 자체가 없다.**
fetch join은 **엔티티 연관**을 한 쿼리로 함께 끌어오는 기법이다(`select p from Post p join fetch p.author`). 그런데 §2 규칙 때문에 `Post`에는 `Member author` 연관이 **없다**(작성자는 원시 `Long`). 조인해서 fetch할 연관이 존재하지 않으니 fetch join을 쓸 수가 없다. 그래서 MEMBER 서비스가 **자기 리포지토리**로 `findByIdIn`을 도는 배치 조회가 유일한 길이고, 이게 곧 "보여주는 것 vs 소유하는 것" 원칙의 실체다.

**(2) 좋아요 수·댓글 수는 "연관"이 아니라 "집계"다.**
fetch join은 연관 엔티티를 **붙여** 오지 개수를 **세지** 않는다. 설령 `Post`에 `@OneToMany` 컬렉션이 있었어도:
* 개수만 필요한데 전체 자식 행을 다 끌어온다(낭비).
* 컬렉션 두 개 이상을 동시에 fetch join하면 카테시안 곱 + `MultipleBagFetchException`.

개수의 올바른 도구는 `GROUP BY ... COUNT`다:
```
select postId, count(*) from PostLike where postId in (:ids) group by postId
```

**요약**: 도메인 경계로 인해 연관이 없어서 fetch join이 **불가능**(1)하고, 필요한 게 연관이 아니라 집계·맵이라 **애초에 fetch join이 맞는 도구가 아니다**(2). → IN 배치 조회 + GROUP BY COUNT.

---

## 6. 한 줄 정리

* "ID로 참조"는 정상이고 우리가 하는 것 = 느슨한 결합.
* 문서가 막는 건 ID 참조가 아니라 **도메인 간 엔티티 연관**(`@ManyToOne 다른도메인`).
* 도메인 **내부** 연관은 허용(선례 `MemberProfile`), 단 FEED는 일관성을 위해 내부도 Long ID.
* 작성자 표시·카운트는 fetch join이 아니라 **서비스 협력 + IN 배치 조회 + GROUP BY COUNT**. fetch join은 (1)대상이 없고 (2)집계엔 부적합해서 안 쓴다.
