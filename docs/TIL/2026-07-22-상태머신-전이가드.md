# TIL — 엔티티에 상태머신 심기: 전이 가드와 이력 보존

> 작성일: 2026-07-22
> 목적: 인증 연주자 신청(PENDING→APPROVED/REJECTED→REVOKED)을 setter가 아니라 **전이 메서드 + 가드**로 모델링한 방법을 이 문서 하나로 이해한다.
> 관련 코드: `domain.verifiedperformer.entity.VerificationApplication`, `VerificationStatus` / 규칙: `docs/DOMAIN-VERIFIED-PERFORMER-STATUTE.md §2`

---

## 1. 문제 — 상태를 setter로 두면 무슨 일이 나나

신청은 여러 상태를 거친다: 대기(PENDING) → 승인(APPROVED)/거절(REJECTED), 승인 후 철회(REVOKED).

만약 `setStatus(status)`로 두면:
- 거절된 신청을 다시 승인하는 **불가능한 전이**를 코드가 막지 못한다.
- "어떤 상태에서 어떤 상태로 갈 수 있나"라는 규칙이 여기저기 흩어진다(서비스마다 if문).
- 잘못된 전이가 **조용히** 성공한다(예외도 없이 데이터가 깨짐).

---

## 2. 해결 — 전이를 "의도가 드러나는 메서드"로, 각 메서드에 가드

setter를 없애고, 허용된 전이마다 이름 있는 메서드를 둔다. 각 메서드는 "지금 상태가 이래야 한다"를 검사(가드)한다.

```java
public void approve(Long adminId, String reason) {
    transitionFrom(PENDING, APPROVED, adminId, reason);   // PENDING 에서만
}
public void reject(Long adminId, String reason) {
    transitionFrom(PENDING, REJECTED, adminId, reason);   // PENDING 에서만
}
public void revoke(Long adminId, String reason) {
    transitionFrom(APPROVED, REVOKED, adminId, reason);   // APPROVED 에서만
}

private void transitionFrom(VerificationStatus required, VerificationStatus next,
        Long adminId, String reason) {
    if (this.status != required) {
        throw new BusinessException(ErrorCode.INVALID_APPLICATION_STATE);  // 불가능한 전이 차단
    }
    this.status = next;
    decide(adminId, reason);   // 처리자·사유·시각 함께 기록
}
```

- 상태 변경 통로가 이 세 메서드뿐이라, **불가능한 전이는 애초에 호출할 수 없거나 가드에서 예외**가 난다.
- 종결 상태(APPROVED·REJECTED·REVOKED)를 다시 처리하려 하면 `INVALID_APPLICATION_STATE`(409)로 **명시적으로 거절**된다. 조용히 깨지지 않는다.
- 상태 전이와 "누가·왜·언제 처리했는지"(decidedBy/decisionReason/decidedAt)가 **한 메서드 안에서 원자적으로** 같이 바뀐다.

```
  [없음] ──신청──▶ PENDING ──approve──▶ APPROVED ──revoke──▶ REVOKED
                     └──reject──▶ REJECTED
  어드민 직접지정 ─────────────▶ APPROVED (신청서 없이)
```

---

## 3. 재신청은 덮어쓰지 않고 새 레코드 — 이력 보존

거절·철회된 회원이 다시 신청할 때, 기존 레코드의 상태를 바꾸지 않는다. **새 레코드**를 만든다.

```java
// 재신청 = 기존 이력 보존 + 새 PENDING 레코드
VerificationApplication.apply(memberId, statement, evidenceUrls);
```

- 과거 신청/심사 이력이 그대로 남는다(감사·추적 가능).
- "언제 거절됐고 다시 언제 신청했나"가 레코드들로 보인다.

---

## 4. 상태로 표현할 수 없는 규칙은 서비스에서 — 활성 유일성

"한 회원은 활성 신청(PENDING/APPROVED)을 하나만 가진다"는 **엔티티 한 개로는 표현 안 되는**(레코드 간) 규칙이라 서비스에서 판정한다.

```java
private void rejectIfActiveApplicationExists(Long memberId) {
    if (repository.existsByMemberIdAndStatus(memberId, PENDING))
        throw new BusinessException(VERIFICATION_ALREADY_PENDING);   // 409-04
    if (repository.existsByMemberIdAndStatus(memberId, APPROVED))
        throw new BusinessException(VERIFICATION_ALREADY_APPROVED);  // 409-05
}
```

교훈: **엔티티 자기 상태 규칙 = 엔티티 안(전이 가드)**, **레코드 간·집계 규칙 = 서비스 안**. 책임을 이렇게 가른다.

---

## 5. 한 줄 정리

- 상태는 `setStatus`가 아니라 **이름 있는 전이 메서드 + 가드**로 바꾼다 → 불가능한 전이를 예외로 차단.
- 종결 상태 재처리는 조용히 무시하지 말고 명시적 에러(`INVALID_APPLICATION_STATE`)로.
- 전이와 함께 처리자/사유/시각을 원자적으로 기록.
- 재신청은 덮어쓰지 말고 새 레코드로 이력 보존.
- 엔티티로 표현 못 하는 레코드 간 규칙(활성 유일성)은 서비스에서.
