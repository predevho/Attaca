# TIL — 커서(keyset) 페이징: offset은 왜 밀리고, 커서는 어떻게 고치나

> 작성일: 2026-07-22
> 목적: FEED 타임라인/댓글에 쓴 커서 페이징을 **이 문서 하나로** 이해한다. offset의 문제, keyset 쿼리, nextCursor 계산, 정렬 안정성(타이브레이크)까지.
> 관련 코드: `domain.feed.repository.PostRepository#findTimeline`, `CommentRepository#findCommentPage`, `domain.feed.service.FeedPostService#getTimeline` / 규칙: `docs/DOMAIN-FEED-STATUTE.md §6`

---

## 1. offset 페이징이 밀리는 이유

`page/size`(offset) 방식은 DB에 "앞의 N개를 건너뛰고 그 다음 M개"라고 시킨다.

```
SELECT * FROM post ORDER BY id DESC LIMIT 10 OFFSET 10   -- 2페이지
```

문제는 **목록이 실시간으로 바뀔 때**다. 사용자가 1페이지를 본 사이 새 글이 1개 올라오면, 모든 항목이 한 칸씩 밀린다.

- 1페이지 마지막 글이 2페이지 첫 줄로 내려가 **중복**으로 다시 보인다.
- 반대로 글이 삭제되면 한 항목이 **건너뛰어져** 안 보인다.

피드처럼 계속 새 글이 쌓이는 목록에서 이 "페이지 밀림"은 눈에 띄는 버그가 된다. (그래서 오프셋은 관리자 목록·연주회 목록처럼 상대적으로 안정적인 데이터에만 쓰고, 피드엔 커서를 썼다.)

---

## 2. 커서(keyset)의 아이디어

"몇 번째부터"가 아니라 **"마지막으로 본 항목 다음부터"** 로 요청한다. 마지막으로 본 항목의 정렬 키 값(여기선 `id`)을 커서로 넘긴다.

```sql
-- 최신순 타임라인: id 내림차순, cursor보다 작은 것부터
WHERE deleted_at IS NULL AND (:cursor IS NULL OR id < :cursor)
ORDER BY id DESC
LIMIT :size
```

- `cursor = null` → 첫 페이지(최신부터).
- `cursor = 마지막 글 id` → 그 글보다 이전 것들.

새 글이 위에 쌓여도 내가 넘긴 커서 아래쪽은 그대로라, **밀림·중복·누락이 없다.**

실제 코드(`PostRepository`):
```java
@Query("select p from Post p "
    + "where p.deletedAt is null and (:cursor is null or p.id < :cursor) "
    + "order by p.id desc")
List<Post> findTimeline(@Param("cursor") Long cursor, Pageable pageable);
```

댓글은 반대로 **오래된순**이라 `id > :cursor ORDER BY id ASC`.

## 3. nextCursor는 어떻게 계산하나 — "size+1 트릭"

다음 페이지가 있는지 알려면 한 개를 더 읽어본다. `size` 대신 **`size+1`** 개를 요청한다.

```java
List<Post> posts = postRepository.findTimeline(cursor, PageRequest.of(0, size + 1));
boolean hasNext = posts.size() > size;          // size+1개가 다 왔으면 다음 페이지 있음
List<Post> page = hasNext ? posts.subList(0, size) : posts;  // 실제 응답은 size개로 자름
Long nextCursor = hasNext ? page.get(page.size() - 1).getId() : null;  // 마지막 항목 id
```

- `nextCursor`가 `null`이면 마지막 페이지.
- 클라이언트는 다음 요청에 이 커서를 그대로 실어 보낸다.
- 응답 형태: `CursorPage<T>(List<T> items, Long nextCursor)`.

**빈 페이지 가드**: `page`가 비면 배치 조회(작성자맵 등)를 건너뛴다(IN 절이 빈 리스트가 되는 것 방지).

---

## 4. 정렬 안정성 — 정렬 키는 유일·단조여야 한다

커서 페이징의 핵심 전제: **정렬 키가 겹치지 않아야 한다.** 겹치면 그 경계에서 항목이 새거나 중복된다.

- FEED는 `id`(단조 증가 PK)로 정렬해서 애초에 겹칠 일이 없다.
- 반면 VERIFIED-PERFORMER는 `createdAt`으로 정렬했는데, **같은 트랜잭션에서 저장된 두 레코드의 `createdAt`이 동일**해 "최신" 판정이 흔들렸다. 리포지토리 테스트가 이걸 잡아서 `id`를 타이브레이크로 추가했다:

```java
// createdAt이 같아도 id로 최종 결정 → 결정적 정렬
findTopByMemberIdOrderByCreatedAtDescIdDesc(Long memberId)
findByStatusOrderByCreatedAtDescIdDesc(status, pageable)
```

교훈: **"createdAt으로 정렬" 같은 비유일 키를 쓸 땐 반드시 유일 키(id)를 타이브레이크로 덧붙인다.** 안 그러면 커서/페이지 경계가 비결정적이 된다.

---

## 5. 한 줄 정리

- offset은 목록이 바뀌면 밀려서 중복/누락 → 피드엔 부적합.
- 커서 = "마지막 본 항목(정렬 키 값) 다음부터". `id < :cursor`(최신순)/`id > :cursor`(오래된순).
- 다음 페이지 유무는 `size+1` 읽어서 판단, `nextCursor` = 잘라낸 마지막 항목 id(없으면 null).
- 정렬 키는 유일·단조여야 한다. 비유일 키(createdAt)면 **id 타이브레이크** 필수.
