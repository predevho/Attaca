# TIL — Java 제네릭(Generics)

> 작성일: 2026-07-05
> 앵커 코드: `com.back.global.common.ApiResponse<T>`

---

## 1. 제네릭이란?

**타입을 "값"처럼 파라미터로 넘기는 문법.** 클래스나 메서드를 정의할 때 구체 타입을 못 박지 않고, 사용하는 시점에 타입을 정한다.

```java
List<String> names = new ArrayList<>();   // 이 리스트는 String 전용
names.add("kim");
String first = names.get(0);              // 캐스팅 없이 String
```

`<String>`이 없다면 `Object`로 다뤄야 하고, 꺼낼 때마다 `(String)` 캐스팅 + 런타임 오류 위험이 생긴다.

---

## 2. 왜 쓰는가 (3가지 이점)

1. **타입 안전성**: 잘못된 타입을 넣으면 **컴파일 시점에** 잡힌다.
   ```java
   List<String> l = new ArrayList<>();
   l.add(123); // 컴파일 에러 — 런타임까지 안 가고 즉시 차단
   ```
2. **캐스팅 제거**: 꺼낼 때 형변환이 필요 없다 → 코드가 깔끔하고 `ClassCastException` 위험 제거.
3. **재사용성**: 같은 로직을 모든 타입에 대해 한 번만 작성. `ApiResponse<Member>`, `ApiResponse<List<Feed>>` 전부 한 클래스로 커버.

---

## 3. 우리 프로젝트 예시 — `ApiResponse<T>`

```java
public class ApiResponse<T> {          // (1) 제네릭 클래스: T는 타입 파라미터
    private final T data;              // (2) 필드 타입으로 사용

    public static <T> ApiResponse<T> success(T data) {  // (3) 제네릭 메서드
        return new ApiResponse<>(true, data, null);
    }
}
```

- `(1)` 클래스 옆 `<T>` → "이 클래스는 T라는 타입을 받아서 동작한다."
- `(2)` 그 T를 필드/파라미터/반환 타입에 사용.
- `(3)` `static` 메서드는 인스턴스 타입 파라미터를 못 쓰므로 **메서드 자체의 `<T>`**를 따로 선언한다. (아래 5절)

사용:
```java
ApiResponse<String> r = ApiResponse.success("hello"); // T = String
ResponseEntity<ApiResponse<Void>> resp = ...;         // T = Void (데이터 없는 응답)
```

`ResponseEntity<ApiResponse<Void>>`처럼 **제네릭 안에 제네릭**도 중첩 가능하다(`GlobalExceptionHandler`에서 실제 사용).

---

## 4. 타입 파라미터 이름 관례

기술적으로는 아무 이름이나 되지만, 관례가 있다:

| 기호 | 의미 |
|---|---|
| `T` | Type (일반 타입) |
| `E` | Element (컬렉션 요소) |
| `K`, `V` | Key, Value (맵) |
| `R` | Return (반환 타입) |
| `N` | Number |

예: `Map<K, V>`, `List<E>`, `Function<T, R>`.

---

## 5. 제네릭 메서드

클래스가 제네릭이 아니어도 **메서드 하나만** 제네릭으로 만들 수 있다. 반환 타입 앞에 `<T>`를 선언한다.

```java
public static <T> T firstOrNull(List<T> list) {
    return list.isEmpty() ? null : list.get(0);
}

String s = firstOrNull(List.of("a", "b")); // T가 String으로 추론됨
```

`static` 팩토리 메서드(`ApiResponse.success`)가 클래스 `<T>`와 별개로 `<T>`를 다시 선언하는 이유가 이것 — **static 문맥에는 인스턴스의 타입 파라미터가 존재하지 않기 때문.**

---

## 6. 경계(Bounded) 타입 파라미터

"아무 타입"이 아니라 **특정 타입의 하위 타입만** 받도록 제한한다.

```java
public static <T extends Number> double sum(List<T> nums) {
    double total = 0;
    for (T n : nums) total += n.doubleValue(); // Number의 메서드 사용 가능
    return total;
}
```

`<T extends Number>` → T는 `Number`나 그 하위(`Integer`, `Double`...)만. 덕분에 T에서 `doubleValue()` 같은 Number 메서드를 호출할 수 있다. (`extends`는 클래스/인터페이스 모두에 사용)

## 7. 와일드카드 `?` — PECS

메서드 "사용처"에서 유연성을 위해 `?`(어떤 타입인지 모름)를 쓴다.

- `<? extends T>` : T 또는 그 하위. **읽기(생산)** 용. → **Producer Extends**
- `<? super T>` : T 또는 그 상위. **쓰기(소비)** 용. → **Consumer Super**

```java
// 읽기만: extends
double total = sum(numbers);                 // List<? extends Number>에서 꺼내 읽기

void addNumbers(List<? super Integer> list) { // Integer를 넣기(소비)
    list.add(1);
}
```

암기법 **PECS**: *Producer-Extends, Consumer-Super.* (꺼내 쓰면 extends, 담으면 super)

> `<T>`(타입 파라미터)와 `<?>`(와일드카드)의 차이: `T`는 "그 타입을 이름 붙여 재사용"할 때, `?`는 "타입을 신경 안 쓰고 유연하게 받기만" 할 때.

## 8. ⚠️ 타입 소거(Type Erasure) — 가장 큰 함정

**제네릭은 컴파일 시점에만 존재하고, 런타임에는 지워진다(erasure).** 컴파일러가 타입 체크를 마친 뒤 `T`를 `Object`(경계가 있으면 그 경계 타입)로 바꿔버린다.

이 때문에 **런타임에 못 하는 것들**이 생긴다:

```java
// 1) T로 배열 생성 불가
T[] arr = new T[10];        // 컴파일 에러

// 2) instanceof에 타입 파라미터 사용 불가
if (obj instanceof T) { }   // 컴파일 에러

// 3) 서로 다른 제네릭이 런타임엔 같은 타입
List<String> a = new ArrayList<>();
List<Integer> b = new ArrayList<>();
a.getClass() == b.getClass();  // true! 둘 다 그냥 ArrayList

// 4) 제네릭 타입 파라미터는 static 필드가 될 수 없음
```

**핵심 교훈**: 제네릭은 "컴파일러가 대신 캐스팅해주고 타입을 검사해주는" 문법 설탕에 가깝다. 런타임 타입 정보가 필요하면 `Class<T>`를 파라미터로 함께 넘기는 식으로 우회한다.

```java
public <T> T parse(String json, Class<T> type) { ... } // type으로 런타임 타입 확보
```

---

## 9. Raw Type은 쓰지 말 것

제네릭 클래스를 타입 없이 쓰는 것(`List list = ...`)을 raw type이라 한다. 하위 호환용으로 허용될 뿐, **타입 안전성이 사라지므로 쓰지 않는다.**

```java
List raw = new ArrayList();  // 경고 발생, 지양
raw.add("a"); raw.add(1);    // 아무거나 들어감 → 위험
```

---

## 10. 한 줄 요약

> 제네릭 = **"타입을 파라미터화해서, 컴파일 시점에 타입 안전성과 재사용을 동시에 얻는 문법."** 단, 런타임엔 타입이 소거된다는 점만 기억하면 대부분의 함정을 피한다.

---

### 참고 — 오늘 프로젝트에서 만난 제네릭
- `ApiResponse<T>` : 응답 래퍼 (제네릭 클래스 + 제네릭 static 팩토리)
- `ApiResponse<Void>` : 데이터 없는 응답
- `ResponseEntity<ApiResponse<Void>>` : 제네릭 중첩
- `List<E>`, `Map<K,V>` : 표준 라이브러리 제네릭
